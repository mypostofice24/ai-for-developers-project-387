# Описание домена и API-контракта календарного бронирования

## Цель

Этот документ фиксирует контекст Design First API-контракта для простого
приложения календарного бронирования. TypeSpec-спецификация в `spec/main.tsp`
задает внешнее поведение приложения для владельца календаря и гостей.

## Домен

- `Owner`: один заранее заданный профиль владельца календаря. В приложении нет
  регистрации, авторизации и API для управления владельцем.
- `EventType`: тип встречи, который создает владелец. Содержит id, название,
  описание и длительность в минутах.
- `Slot`: вычисленный свободный интервал для выбранного типа события. Слоты не
  создаются владельцем вручную.
- `Booking`: бронирование гостя для типа события и выбранного слота.

## Поведение

- Владелец может создавать типы событий.
- Владелец может смотреть предстоящие бронирования всех типов событий в одном
  списке.
- Гость может смотреть публичные типы событий.
- Гость может выбрать тип события и посмотреть свободные слоты на ближайшие 14
  дней.
- Гость может создать бронирование для свободного слота.
- Доступные слоты формируются каждый день с 09:00 до 18:00 по локальному времени
  приложения.
- Шаг сетки слотов равен `EventType.durationMinutes`.
- Один временной интервал нельзя забронировать дважды, даже если бронирования
  относятся к разным типам событий.

## API-контракт

Контракт описан в TypeSpec с https://typespec.io/ в формате REST + JSON с
генерацией OpenAPI. Основная спецификация находится в `spec/main.tsp`.

Ожидаемые endpoints:

- `GET /admin/event-types`
- `POST /admin/event-types`
- `GET /admin/bookings`
- `GET /event-types`
- `GET /event-types/{eventTypeId}/slots`
- `POST /bookings`

Ожидаемые ошибки:

- `404 Not Found`, если указанный тип события не существует.
- `409 Conflict`, если запрошенное бронирование пересекается с существующим.
- `422 Unprocessable Entity`, если не проходит бизнес-валидация: например, слот
  находится вне 14-дневного окна, вне доступности 09:00-18:00 или не совпадает с
  шагом длительности типа события.

## Проверка покрытия

- Владелец создает типы событий: `POST /admin/event-types`.
- Владелец смотрит типы событий в админской части: `GET /admin/event-types`.
- Владелец смотрит предстоящие бронирования: `GET /admin/bookings`.
- Гость смотрит типы событий: `GET /event-types`.
- Гость смотрит свободные слоты для выбранного типа события:
  `GET /event-types/{eventTypeId}/slots`.
- Гость создает бронирование: `POST /bookings`.
- Двойное бронирование предотвращается через `409 Conflict`.
- Окно записи и правила генерации слотов зафиксированы в документации TypeSpec и
  ошибках валидации.
