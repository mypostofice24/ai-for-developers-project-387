# Интеграционные сценарии

Эти сценарии проверяют приложение как связку отдельного frontend и реального backend.
Prism mock в e2e не используется.

## Guest booking happy path

1. Гость открывает главную страницу.
2. Гость видит список типов событий, полученный из backend.
3. Гость выбирает тип события на 30 минут.
4. Страница бронирования загружает доступные слоты из backend.
5. Гость выбирает первый доступный слот.
6. Гость вводит имя и email.
7. Гость подтверждает запись.
8. Интерфейс показывает успешное создание записи.

## Занятый слот

1. После успешного бронирования тот же `eventTypeId + startAt` повторно отправляется в backend.
2. Backend возвращает `409` и код `BOOKING_CONFLICT`.
3. Это подтверждает, что занятость слота проверяется сервером, а не только frontend.

## Admin visibility

1. После создания бронирования открывается `/admin/bookings`.
2. Новая запись отображается в списке предстоящих бронирований.
3. Проверяются имя и email гостя из e2e-сценария.
